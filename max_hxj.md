# 苹果缺陷检测项目总结 (Apple Defect Detection)

本项目旨在为工业检测场景（如苹果或相关产品表面的异物、过切等缺陷）提供一套完整的自动化视觉检测解决方案。在这个项目中，我们经历了从最基础的检测方案，到引入图像切片技术，再到最终采用基于 Transformer 架构的最新目标检测算法，并且通过数据集的反复清洗与优化，最终达到极其优秀的检测效果的过程。

## 技术架构

我们的系统采用了模块化的多 Pipeline 架构，能够随时切换不同的底层算法引擎：

1. **Pipeline A (基于 YOLOv8)**
   - 早期我们使用了经典的 YOLOv8。由于工业相机拍摄的图片分辨率极大（如 5472×3648），如果直接把这么大的图片塞进 YOLO，图片会被严重压缩（比如压缩到 640x640），导致原本就微小的缺陷（例如微小异物）直接“消失”在像素中，检测效果惨不忍睹。
   
2. **Pipeline B (基于 Anomalib)**
   - 我们也尝试了无监督异常检测（Anomaly Detection），这种方法只需要正常样本进行训练，在遇到和正常样本不一样的图案时就会报警。但无监督方法的误报率较高，且很难精确分类具体的缺陷类型。

3. **Pipeline C (基于 RF-DETR + 切片技术)** —— **当前核心主力**
   - 这是我们最终敲定并取得突破性进展的方案。RF-DETR（Real-Time DEtection TRansformer）是新一代的目标检测模型。为了解决大图下小缺陷丢失的问题，我们为它搭配了**图像切片（Slicing/Tiling）技术**。
   - **工作流**：先把 5000 多像素的大图，按照 640x640 的大小像切蛋糕一样切成几十块（相互之间保留 25% 的重叠防止缺陷被切断）；然后让模型对每一个小块进行检测；最后，通过 NMS（非极大值抑制）算法，把各个小块的检测结果重新拼回大图的坐标系中，去掉重复的框。

除了这三大管线，外围还配备了基于 `FastAPI` 的后台接口和基于 `Gradio` 的可视化工作台（Training Workbench），能够非常方便地通过网页点击按钮来启动训练、查看实时日志并测试单张图片的检测效果。

## 技术决策与为什么这么做

- **为什么必须使用图像切片？**
  - **教训**：我们曾尝试直接把未切片的大图丢给模型训练。结果显示，对于稍大一些的缺陷（如过切），模型还能勉强认出，但对于微小的“异物”缺陷，召回率直接跌到了 0%。因为 5000 像素的图片缩小到 512 像素后，几个像素的异物连一个像素都不剩了。
  - **决策**：强制引入切片机制。虽然这会增加推理时间（一张大图要推理七八十次），但在工业质检中，准确率和不漏检是第一位的。

- **为什么切换到 RF-DETR，而且选用 Medium 变体？**
  - YOLOv8 虽然快，但在密集小目标和复杂背景下的表现有时不如基于注意力机制的 Transformer 模型。RF-DETR 结合了 Transformer 的全局感知能力和实时检测的速度。
  - 默认的 Nano 或 Small 模型参数量较小，为了进一步榨干数据集的潜力，我们在系统中设计了 `--variant` 参数来支持动态加载模型大小，最终在 Medium 模型上取得了 `89.5%` 的 F1 分数（召回率高达 94.4%）。

- **为什么反复清洗和对齐数据集？**
  - **坑点**：我们在训练中发现，无论模型怎么调，准确率都卡在一个瓶颈上。经过分析，发现原始数据集存在“污染”——有些缺陷标注不准，有些图片边缘有干扰黑边。
  - **决策**：我们编写了专门的脚本，把图片重新旋转对齐、裁剪掉黑边，生成了 `batch1_coco_p03_aligned_crop2` 这样极其纯净的数据集。**数据决定了模型表现的上限**，换用干净数据集后，精确率瞬间从低谷飙升到了 85%。

## 经验教训与避坑指南

1. **别迷信大模型，数据才是王道**
   - 当模型表现差时，不要第一时间去修改模型结构或调参。打开数据集看一看！我们曾经对着表现不佳的日志发愁，结果发现是由于数据集本身存在黑边干扰和标注错误。清洗数据（Crop & Align）带来的提升，远比换一个更复杂的模型来得大。

2. **大图小目标检测，切片是唯一的出路**
   - 工业检测动辄几千万像素，任何号称能直接处理这种分辨率且不丢失小目标的神奇网络目前都是不现实的（显存也撑不住）。老老实实做切片（Slicing），并在切片边缘做好重叠（Overlap），最后再映射回全局坐标做 NMS，这是一套非常成熟且必须掌握的组合拳。

3. **系统设计要留有余地（解耦）**
   - 我们的系统设计了 `BasePipeline`，这让我们可以非常轻松地把 YOLO 换成 Anomalib，再换成 RF-DETR，而不需要去改动后台 API 和前端页面的核心代码。前端只需要多传一个 `--pipeline c` 参数，后台就能无缝对接。

4. **PowerShell 中的日期格式坑**
   - 在写自动化脚本时，如果你要在 PowerShell 里用 `Get-Date -Format` 生成带有日期后缀的文件，千万记得 `yyyy-MM-dd` 这种是占位符。直接写死数字会被识别为字面量报错，这点小坑浪费了我们不少调试时间。

## 结语
这个项目完整地展现了一个工业 AI 视觉系统的演进路线：从最初的直接检测碰壁，到引入切片技术解决分辨率问题，再到清洗数据突破准确率瓶颈，最后通过引入前沿的 RF-DETR 架构达到极其优异的生产级表现。这是一次非常有价值的工程实践。
